Professional Self-Assessment

Completing the Computer Science program at Southern New Hampshire University has been a foundational experience that shaped both my technical skills and how I think about software professionally. When I started in 2023, I had a general interest in technology but limited real experience building software systems. That changed over the course of the program. Through courses covering data structures, algorithms, full-stack development, secure coding, and software testing, I built the skills and confidence to approach real engineering problems. The capstone ePortfolio brings all of that together through enhancements to one real-world application, demonstrating growth across software engineering, algorithms, and database design.

Collaborating in a Team Environment

CS 250: Software Development Lifecycle was where I first learned how software is actually built in a professional setting. Working through the Agile development process, writing user stories, and participating in sprint planning taught me that writing code is only part of the job. Good code has to be readable and structured in a way that other developers can work with it. That idea shaped how I approached this capstone. The service layer I built separates business logic from the HTTP layer so that any developer on the team can change one part without needing to understand everything else. That kind of structure is not just about personal preference -- it is a form of collaboration built into the architecture.

Communicating with Stakeholders

CS 250 also reinforced that engineers have to explain their decisions to people who are not engineers. Learning to write user stories and present trade-offs in plain language helped me understand that technical ability without communication is not enough. In the code review I produced for this capstone, I explained my planned enhancements in terms of what they would improve about the application, not just the technical details of how they worked. Being able to frame technical decisions in terms of outcomes is a skill I want to keep developing as I move into my career.

Data Structures and Algorithms

CS 260: Data Structures and Algorithms gave me the foundation I actually used in this capstone. Understanding how hash maps and linked lists work at the implementation level made it possible to build the custom LRU cache in the Travlr API from scratch, rather than relying on a library. I used a doubly linked list for fast move-to-front operations and a hash map for O(1) key lookups -- the same structure used in real caching systems. The MongoDB aggregation pipeline I built for filtering trips by price range required similar thinking: structuring a sequence of operations to compute, filter, sort, and paginate data while minimizing unnecessary work.

Software Engineering and Databases

CS 465: Full Stack Development I introduced the MEAN stack and gave me the foundation for this capstone artifact. The enhancements I made through CS 499 address what a single course could not fully cover: production-quality backend architecture. Adding a layered structure with routes, controllers, and services, centralized error handling, input validation middleware, and role-based access control represents the difference between a course project and something built to professional standards. On the database side, adding strict schema validation, unique constraints, and enum enforcement on user roles showed me that database integrity has to be designed in from the start. Embedding user roles in JWT payloads to avoid per-request database lookups was a decision that came directly from thinking about how database design affects performance.

Security

CS 405: Secure Coding gave me a way to think about security that I have not stopped using since. The most important idea from that course was defense in depth: no single layer should be the only thing protecting the system. In this capstone I applied that directly. Input is validated by middleware before it reaches the controller. Constraints are enforced again at the Mongoose schema layer before data is written. Write routes require both JWT authentication and role-based authorization. Each layer is designed assuming the previous one might be bypassed. That approach to security, anticipating failure and building protection at every level, is what Course Outcome 5 is really asking for.

Portfolio Summary

All three enhancements in this portfolio build on the same artifact and they are connected. The software engineering enhancement introduced the architecture that makes the code maintainable. The algorithms enhancement added the caching and query system that makes the API fast and flexible. The database enhancement tightened the schema and added access control that makes the system secure. These are not separate demonstrations -- they depend on each other. A clean architecture is easier to secure. Secure access control depends on a well-designed schema. Efficient queries depend on a service layer that can be reasoned about clearly. Together, they show that I can design, build, and reason about software at a professional level.

Enhancement One
Category 01 -- Software Design and Engineering

Backend Architecture Refactor

Artifact Description

The artifact for this enhancement is the Travlr Getaways full-stack web application, originally developed in CS-465: Full Stack Development I. This project uses the MEAN stack and includes a web interface for managing travel data and an API backend connected to a MongoDB database. The application was completed in the final weeks of CS-465.

Justification and Improvements

I selected this artifact because it represents a realistic, industry-relevant backend system with meaningful room for architectural improvement. The original implementation had routes and controllers separated, but several gaps remained that would be problematic in a professional environment.

The specific improvements made for this enhancement include: a new app_api/services/tripService.js module that holds all database interaction logic, separating it from HTTP handling; a dedicated app_api/middleware/errorHandler.js that differentiates between JWT authentication errors, Mongoose validation errors, and general server errors; and an input validation middleware that checks all required fields and types before requests reach the controller. Together these changes move the backend from a working prototype toward industry-level layered architecture.

Course Outcomes

This enhancement meets Course Outcome 4 (using well-founded techniques to implement solutions that accomplish industry-specific goals) and Course Outcome 5 (developing a security mindset that anticipates adversarial exploits). The validation middleware prevents malformed data from reaching the database. The centralized error handler ensures internal server details are never leaked to the client in production.

Reflection

The most valuable thing I learned was how much cleaner a controller becomes when it does not own the database logic. Before the refactor, each controller was mixing HTTP handling, database queries, and error formatting together. After extracting the database calls into the service layer and routing errors through a single middleware, each layer has a single, clear responsibility. The main challenge was rethinking the error flow -- Express's four-parameter error middleware requires that every async handler passes errors to next(err) rather than catching and responding inline. Updating every controller consistently was a straightforward but important change to get right.

Enhancement Two
Category 02 -- Algorithms and Data Structures

LRU Cache, Pagination, and Filtering

Artifact Description

The artifact for this enhancement is the same Travlr Getaways application from CS-465, specifically the backend API layer responsible for querying and returning trip data. The original implementation retrieved every trip document from the database in a single unfiltered query and returned all results at once, with no caching.

Justification and Improvements

I selected this artifact because the trip listing endpoint is the most data-intensive part of the application and represented a clear opportunity to apply algorithmic thinking. The improvements include a custom LRU cache implemented from scratch using a doubly linked list and a hash map, skip-offset pagination, field-based sorting, case-insensitive resort filtering, and numeric price-range filtering via MongoDB aggregation pipeline.

The LRU cache was the most algorithmically interesting piece. Most recently accessed entries are kept at the front of the doubly linked list and the least recently used entry at the tail is evicted when capacity is reached. Both get and put operations run in O(1) time. The price-range filter required an aggregation pipeline to strip formatting characters from the perPerson string field and cast it to a double for numeric comparison, without changing the schema.

Course Outcomes

This enhancement meets Course Outcome 3 (designing computing solutions using algorithmic principles while managing trade-offs) and Course Outcome 4. The skip-offset pagination algorithm trades simplicity for performance at very large scale -- cursor-based pagination would perform better on massive datasets but is significantly more complex to implement. Documenting that trade-off demonstrates awareness of algorithmic design decisions rather than just implementing a solution.

Reflection

The biggest challenge was the price filtering. Because perPerson is stored as a string, I could not add a simple numeric comparison filter. My first instinct was to add a separate numeric field to the schema, but that would require a migration and create a maintenance burden. Using the aggregation pipeline to compute the numeric value at query time was the cleaner solution. Running the count and data pipelines in parallel using Promise.all was a small but meaningful efficiency improvement that I had not done before.

Enhancement Three
Category 03 -- Databases

Schema Validation and Role-Based Access Control

Artifact Description

The artifact is the database architecture and backend API for the Travlr application, a travel booking website originally created during CS-465. It is built using the MEAN stack, specifically relying on MongoDB and Mongoose for data modeling, storage, and retrieval.

Justification and Improvements

I selected this artifact because it provides a practical demonstration of database design, data integrity, and security. Originally, the database schemas were very basic and lacked strict validation, leaving the application vulnerable to bad data and unauthorized modifications.

I improved the artifact by enforcing strict schema-level constraints within Mongoose -- unique identifiers on trip codes, maxlength properties on text fields, and enum validators. I also implemented a database-backed Role-Based Access Control system. By adding a strict role field (restricted to either user or admin) to the User schema and embedding this role into authentication JWTs, only authorized administrators can execute POST, PUT, and DELETE operations on the trips collection.

Course Outcomes

This enhancement meets Course Outcome 4 (using well-founded techniques in database implementation to accomplish industry goals) through the robust schema design and query validations. It also strongly meets Course Outcome 5 (developing a security mindset) because I anticipated adversarial exploits and mitigated design flaws by locking down the database against unauthorized modifications.

Reflection

Reflecting on this process, I learned how critical it is to treat the database schema as the final line of defense for data integrity. Relying solely on API-level validation is not enough -- the database itself must reject malformed data. The main challenge was figuring out how to check a user's role efficiently without a database query on every request. I solved this by embedding the role directly into the JWT payload at login time, so the RBAC middleware can read it instantly from the token. This improved my understanding of how database architecture intersects with application performance and security.
